2 Syntactic FormsðŸ”—

To simplify the composition of effects and the definition of handlers, the library provides standard syntactic sugar.

syntax

(do clause ...)

 
clause = [id <- expr]
  | expr
A monadic sequencing macro similar to Haskell’s do-notation. It expands into nested calls to >>=.

  • [id <- expr]: Executes expr. The result is unpacked and bound to id for the scope of the remaining clauses. Binding happens through match, which means that id must take the form of a match clause.

  • expr: Executes expr for its side effects, ignoring its result (binding it to _), and proceeds to the next clause.

The final expression in the sequence dictates the return value of the whole do block.

Examples:
> (struct ask effect-desc (question))
> (define (ask-user q) (effect (ask q) return))
> (define survey-computation
    (do [name <- (ask-user "What is your name?")]
        [age  <- (ask-user "What is your age?")]
        (return (format "~a is ~a years old." name age))))
> (with-effect-handlers
    ([(ask "What is your name?") (return "Alice")]
     [(ask "What is your age?") (return 30)])
    survey-computation)

"Alice is 30 years old."

syntax

(with-effect-handlers (handler-clause ...) body ...)

 
handler-clause = [pattern handler-body ...+]
  | [pattern handler-body ... (abort result-expr)]
Runs the code in body ... within a context that handles effects matching the provided patterns. This macro expands to a call to run.

The handler-clauses use match syntax to destructure the effect descriptor. The implicit effect object (containing the descriptor and continuation) is available to the macro expansion but hidden from the user syntax to streamline code.

There are two types of handler clauses:

  • **Resuming Handlers**: By default, the result of the handler-body is passed to the effect’s continuation k. The computation then proceeds via >>=.

  • **Aborting Handlers**: If the clause ends with (abort result-expr), the computation is short-circuited. The continuation k is discarded, and result-expr (wrapped in pure) becomes the final result of the handled block.

If an effect is raised that does not match any clause, an error is raised.

Examples:
> (struct log effect-desc (message))
> (struct crash effect-desc (reason))
> (define (log-msg m) (effect (log m) return))
> (define (fail r) (effect (crash r) return))
> (with-effect-handlers
    ([(log m)
      (displayln m)
      (return (void))]
  
      [(crash reason)
      (displayln (format "System aborted: ~a" reason))
      (abort 'failed)])
  
    (do (log-msg "Starting process...")
        (log-msg "Process running...")
        (fail "Critical Error")
        (log-msg "This will never be printed")))
